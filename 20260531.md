## Metric Clarification

### Pearson Delta20

Pearson Delta20 measures correlation on the top 20 most changed genes, usually selected by absolute real delta:

```text
real_delta = real_perturbed_mean - control_mean
pred_delta = pred_perturbed_mean - control_mean
top20 = genes with largest abs(real_delta)
Pearson(real_delta[top20], pred_delta[top20])
```

This is not necessarily the same as DEG metrics. It is a fixed top-20 effect-size subset.

### DS

The previous global DS value for MyFlow was misleading. MyFlow showed `DS=1.0` because the earlier calculation was effectively too global and averaged across all perturbations in a way that made it look perfect.

Per-condition DS20 was much lower, around `0.694`, so `DS=1.0` should not be interpreted as real biological accuracy.

### DEG PCC / DEG R2 / DE-Spearman

These should be computed per perturbation condition using DE genes selected from the true data, then averaged across perturbations.

Correct principle:

```text
For each perturbation:
  1. compare real perturbed cells vs matched control cells
  2. select DE genes from real data only
  3. compute real_delta and pred_delta on those DE genes
  4. compute PCC, R2, Spearman
Average metrics over perturbations
```

The prediction must not be used to choose DE genes.

## What STRING PPI Means

STRING PPI itself is not a perturbation-response graph.

It is a gene/protein-level biological network:

```text
gene/protein A interacts with or is functionally associated with gene/protein B
```

When the perturbation is a gene perturbation, this can be used as a perturbation-gene relationship prior:

```text
if perturbation = knockdown gene A
and perturbation = knockdown gene B
and A and B are close in STRING
then their perturbation effects may be related
```

So the precise wording is:

```text
STRING PPI is not direct perturbation-response truth.
It is a perturbation-gene relationship prior when perturbations are genes.
```

TxPert uses it this way:

```text
nodes = perturbation genes
edges = STRING / GO relationships among perturbation genes
message passing = perturbation embedding propagation
```

## TxPert Prior vs MyFlow Current Prior

### TxPert

TxPert graph is perturbation-level:

```text
perturbation gene A <-> perturbation gene B
```

It learns perturbation embeddings over a graph of perturbed genes.

### MyFlow Current GO Response Prior

MyFlow current GO response branch is output-gene response-level:

```text

pair(output gene embedding z_i, perturbation gene embedding z_p) -> rho_i
```

It is not equivalent to TxPert perturbation graph.

## Can MyFlow Use TxPert STRING PPI?

Yes, but not by simply replacing the current GO response graph file.

If we directly feed STRING PPI into current `GOResponsePriorEncoder`, it becomes:

```text

```

That is not the same as TxPert's:

```text

```

The correct MyFlow design should add a separate perturbation graph branch:

```text

  -> exact symbol lookup
  -> gene2vec initial embedding
  -> STRING / GO perturbation graph propagation
  -> perturbation embedding

```

This should replace nearest-neighbor matching against output genes.

## Is the Pairwise rho Design Reasonable?

The proposed pairwise prior is:

```text
z_p = perturbation gene embedding
z_i = output gene embedding

rho_i = MLP([z_p, z_i, z_p * z_i, abs(z_p - z_i)])
```

This produces:

```text
rho shape = [batch, num_output_genes, rho_dim]
```

Interpretation:

```text
rho_i is a learned pairwise feature for:

perturbation gene p -> output expression gene i
```

This is reasonable, but it must be described carefully.

It is not a hard biological regulatory edge. It does not mean:

```text
PTPN1 definitely regulates GeneA
```

It means:

```text
Given the embedding of PTPN1 and the embedding of GeneA,
let the model learn a gene-specific prior feature that may help decide
how PTPN1 should affect GeneA's velocity.
```

So the precise wording should be:

```text
learned gene-pair prior representation
```

or:

```text
perturbation-to-response gene prior feature
```

not:

```text
explicit biological regulation graph
```

### Why It Is Reasonable

First, the perturbation object and output object are both genes, so they can be represented in a shared gene embedding space.

Second, gene2vec / GO-derived embeddings may contain functional similarity, pathway context, or co-occurrence information.

Third, the pairwise construction is more fine-grained than GEARS / TxPert style perturbation embeddings.

GEARS / TxPert mostly do:

```text
PTPN1 -> one perturbation embedding
```

Then the perturbation embedding affects the whole expression vector.

The pairwise MyFlow branch does:

```text
PTPN1 + GeneA -> GeneA-specific prior
PTPN1 + GeneB -> GeneB-specific prior
PTPN1 + GeneC -> GeneC-specific prior
```

This is closer to a response-gene-specific prior.

### Limitation

The limitation is important:

```text
similarity between z_p and z_i does not prove that perturbing p regulates gene i
```

For example, `PTPN1` may affect a signaling pathway whose downstream expression genes are not necessarily close to `PTPN1` in gene2vec or GO space.

Therefore this pairwise rho should be treated as:

```text
weak prior + learnable interaction
```

not as a true causal gene regulation prior.

### How to Make rho More Biological Later

The pair feature can later include explicit biological relationship features:

```text
rho_i = MLP([
  z_p,
  z_i,
  z_p * z_i,
  abs(z_p - z_i),
  GO_similarity(p, i),
  STRING_distance(p, i),
  same_pathway(p, i),
  TF_target_edge(p, i)
])
```

This would make the branch more directly represent:

```text
perturbation gene and output gene relationship prior
```

For now, using only the embedding pair is still acceptable as a first version, but it should be evaluated by ablation.

Recommended ablation:

```text
MyFlow raw
MyFlow + corrected exact perturbation embedding
MyFlow + pairwise rho
MyFlow + pairwise rho + velocity modulation
```

The paper/story should emphasize:

```text
prior-guided velocity field
```

not merely:

```text
graph-enhanced perturbation embedding
```

## 2026-05-31 晚间更新：Combined Graph Encoding 实现

### 核心架构变更

将 GO prior 分支从"两个独立 gene2vec 表"改为"combined 统一图编码"：

**Combined Gene Table**：输出基因 (HVG 2000) + 扰动基因 (68) 合并为一个 gene2vec 表 (2055 genes)。
所有基因共享同一个 GO 图，经过相同的图消息传递，保证输出基因和扰动基因在同一 latent space。

**数据流**：
```
combined gene2vec table (2055 genes)
  → GO graph message passing (shared)
  → split: z_i = graph_nodes[output_indices]     # (2000, dim)
           z_p = graph_nodes[perturb_indices]     # (68, dim)
  → attention-weighted aggregation: z_p_agg       # (B, 2000, dim)
  → pair features: [z_i, z_p_agg, z_i*z_p_agg, |z_i-z_p_agg|]  # (B, 2000, 4*dim)
  → MLP → rho                                    # (B, 2000, rho_dim)
```

### 修改的文件

## 2026-05-31 深夜更新：Cosine Loss + Gradient Clipping

## 2026-06-01 傍晚更新：新增 DEG Overlap 指标

### 为什么要加这个指标

目前在 Replogle LOCO 上，MyFlow 已经在 MSE、Pearson Δ、DS、DE-Spearman 四个指标上超过 TxPert，但 PCC (0.340 vs 0.422) 和 Pearson Δ20 (0.598 vs 0.773) 仍有差距。

DEG Overlap 可以诊断这个差距的来源：
- **Precision (预测的 DEG 中多少是真实 DEG)**：高 = 模型预测的 top 变化基因是真的
- **Recall (真实 DEG 中多少被模型正确预测)**：高 = 模型不遗漏重要变化基因
- **F1 / Jaccard**：综合指标

这能回答一个关键问题：**MyFlow 预测表达谱整体方向是对的（Pearson Δ 更好），但在 top 变化基因的排序上为什么不如 TxPert？**

可能情况：
| 现象 | 含义 | 对策 |
|------|------|------|
| Precision 高、Recall 低 | 模型预测保守，只抓到部分真实 DEG | 增大 endpoint/cosine weight，让模型更大胆 |
| Precision 低、Recall 高 | 模型预测了太多假 DEG，噪声大 | 增加正则化，降低 noise |
| 两者都低 | 模型在 DEG 空间完全不准 | 需要架构改进 |
| Precision 高、Recall 高 | 但 PCC/Δ20 仍低，说明排序不对 | 加排序 loss |

### 具体实现

在 `scripts/train_myflow_loco_new.py` 中新增 `cal_deg_overlap_metrics()` 函数：

- 对每个 perturbation，从 pred_delta 中用 robust z-score 选出预测的 DEG
- 与真实 DEG 做交集，计算 precision/recall/F1/Jaccard
- 与现有 `identify_degs()` 使用完全相同的阈值 (z-score > 2.0)，确保可比
- 每个 condition 的结果写进 `deg_per_condition_{run_label}.json`
- 全局平均写进 `metrics_summary_{run_label}.json`

### 记录格式

```
DEG Overlap => Precision: X.XXXX, Recall: X.XXXX, F1: X.XXXX, Jaccard: X.XXXX
```

同时每 condition 记录了 n_real_degs / n_pred_degs / n_overlap_degs，便于分析哪些 perturbation 预测得好、哪些差。

## 2026-06-01 会话 3：MyFlow mean-delta 训练目标增强

### 背景判断

TxPert 的训练目标里没有显式的 distribution loss、MMD/Wasserstein、DEG loss 或 condition mean prior。实际代码是 endpoint reconstruction：

```python
mse_loss = F.mse_loss(x_hat, x, reduction="mean")
cos_loss = 1 - F.cosine_similarity(x_hat, x, dim=-1).mean()
return self.mse_weight * mse_loss + (1 - self.mse_weight) * cos_loss
```

当前 wrapper 中 `mse_weight=1.0`，所以 TxPert 实际是纯 MSE endpoint regression。它效果好的主要原因不是隐藏加载权重，也不是偷偷用了 DEG 目标，而是它的目标天然更贴近 mean/delta/DEG 指标：

```text
control cell + perturbation embedding -> perturbed expression
```

这个直接 endpoint MSE 会强烈推动模型学习每个 perturbation condition 的条件均值。mean Pearson、delta Pearson、DEG PCC/F1 等指标都主要依赖 condition mean 或 condition delta，因此 TxPert 在这些指标上有天然优势。

MyFlow 的主目标是 velocity matching：

```text
loss = mean((v_t - u_t)^2)
```

它更偏向学习从 control distribution 到 target distribution 的流，而不是直接优化 condition mean/delta。之前 Replogle 脚本中 `endpoint_mse_weight=0.0`，导致 Replogle 上几乎没有直接 endpoint 监督，这对 DEG 和 mean delta 指标非常不利。

### 设计目标

这次修改不是把 MyFlow 改成纯 mean predictor，而是在保留 flow matching / distribution modeling 的基础上，增加三个弱监督项，让模型更重视 condition-specific response：

1. endpoint MSE：直接约束终点表达。
2. condition mean delta loss：直接约束条件均值变化方向和幅度。
3. high-delta gene weighting：对真实变化大的基因加权，提升 DEG 相关指标。
4. delta cosine loss：约束扰动 delta 的方向。

目标是改善：

```text
Pearson delta
Pearson delta top-k
DEG PCC / R2 / EV
DEG precision / recall / F1
condition-specific matching
```

同时需要监控是否损害原本的 distribution 指标：

```text
MMD
gene-wise Wasserstein
diversity ratio
coverage
variance/covariance similarity
```

### Solver 修改：`myflow/solvers/_otfm.py`

新增 solver kwargs：

```text
condition_mean_delta_weight
high_delta_endpoint_weight
high_delta_eps
```

保留已有：

```text
endpoint_mse_weight
cosine_loss_weight
condition_combined_loss_weight
```

新增 endpoint 加权逻辑：

```python
endpoint_sq_err = (x1_pred - target) ** 2
true_delta_abs = abs(mean(target - source, axis=0))
gene_weights = true_delta_abs / (mean(true_delta_abs) + eps)
gene_weights = 1 + high_delta_endpoint_weight * gene_weights
endpoint_loss = mean(endpoint_sq_err * gene_weights)
```

新增 batch-level condition mean delta loss：

```python
x1_pred = x_t + (1 - t) * v_t
mean_delta_pred = mean(x1_pred - source, axis=0)
mean_delta_true = mean(target - source, axis=0)
condition_mean_delta_loss = mean((mean_delta_pred - mean_delta_true)^2)
```

注意：当前 `TrainSampler` 每个 batch 采样的是一个 target condition，因此 batch 内 mean delta 就是该 condition 的 mean delta，不需要再按多个 condition 分组。

高变化基因加权也应用在 condition mean delta loss 上：

```text
mean_delta_sq_err *= gene_weights
```

### Norman 脚本适配：`scripts/train_myflow_norman_additive.py`

新增命令行参数：

```text
--condition-mean-delta-weight
--high-delta-endpoint-weight
--cosine-loss-weight
```

默认值：

```text
endpoint_mse_weight = 1.0
condition_mean_delta_weight = 1.0
high_delta_endpoint_weight = 1.0
```

### 预期效果和风险

合理预期：

```text
DEG PCC / DEG F1 / delta Pearson 应该上升；
Replogle 的提升可能比 Norman 更明显，因为 Replogle 原先 endpoint_mse_weight 是 0。
```

不能保证：

```text
不保证直接超过 TxPert。
TxPert 本质上很像强 condition mean predictor，因此它在 mean/DEG 指标上仍然可能有优势。
```

主要风险：

```text
如果 mean-delta / endpoint loss 太强，MyFlow 可能更靠近 condition mean，导致 diversity/MMD/coverage 优势下降。
```

因此跑完后不能只看 DEG，还必须同时看 distribution metrics：

```text
如果 DEG 提升且 diversity ratio 仍显著高于 TxPert：
  说明 MyFlow 既能保留细胞分布，又能更好捕捉 perturbation direction。

如果 DEG 提升但 diversity 明显下降：
  说明 mean supervision 太强，需要调小 condition_mean_delta_weight 或 high_delta_endpoint_weight。

如果 DEG 没提升：
  说明问题不只是 loss 权重，可能是 condition encoder / perturbation prior / split 难度 / normalization 的问题。
```

### 推荐调参顺序

第一版使用：

```text
endpoint_mse_weight = 1.0
condition_mean_delta_weight = 1.0
high_delta_endpoint_weight = 1.0
cosine_loss_weight = 0.1
```

如果 diversity 掉太多：

```text
condition_mean_delta_weight -> 0.5
high_delta_endpoint_weight -> 0.5
```

如果 DEG 仍然弱：

```text
condition_mean_delta_weight -> 2.0
high_delta_endpoint_weight -> 2.0
cosine_loss_weight -> 0.2
```

建议固定调参规则，不能根据测试集逐条件挑参数。可以用 validation 或统一规则选择 alpha/weight。
